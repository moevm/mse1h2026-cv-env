# mse-template
## О проекте
### Постановка задачи
Создать единую веб-платформу для работы с датасетами компьютерного зрения, которая решает проблему разрозненности инструментов на этапах подготовки и обучения моделей.

### Требования
### **Функциональные требования**
#### **1. Работа с датасетами** 
##### **1.1 Загрузка датасета**  
+ Пользователь должен иметь возможность загрузить датасет из локальной директории или архива ***.zip***;


+ При загрузке датасета система должна проверить корректность структуры папок ***images/*** и ***labels/***;


+ Если структура датасета некорректна, система должна уведомить пользователя об ошибке загрузки;


+ После успешной загрузки система должна отобразить структуру датасета и количество изображений;



##### **1.2 Разбиение датасета**


+ Пользователь должен иметь возможность задать пропорции разбиения датасета перед запуском процедуры.



##### **1.3 Версионирование**
+ При внесении изменений в датасет (разметка, предобработка) система должна создавать новую версию датасета.


+ Пользователь должен иметь возможность просмотреть список версий датасета.


+ Пользователь должен иметь возможность откатиться к любой предыдущей версии датасета.


#### **2. Разметка данных**
##### **2.1 Bounding Box**
+ После открытия изображения система должна отобразить его в интерфейсе разметки.


+ Пользователь должен иметь возможность создать *bounding box* с указанием класса объекта.


+ Пользователь должен иметь возможность редактировать или удалить созданную рамку.


+ После сохранения разметки система должна создать или обновить ***.txt*** файл в формате ***YOLO***.


+ Если на изображении отсутствуют объекты, система должна создать пустой ***.txt*** файл разметки.


##### **2.2 Работа с видео**
+ При загрузке видео система должна автоматически разбить его на отдельные кадры.


+ Пользователь должен иметь возможность выполнять разметку кадров аналогично разметке изображений.

##### **3. Предобработка**
+ Пользователь должен иметь возможность выбрать параметры предобработки датасета (ресайз, аугментации, нормализация).


+ После подтверждения параметров система должна применить выбранные преобразования ко всему датасету.


+ При завершении предобработки система должна сохранить обновлённую версию датасета.



##### **4. Обучение модели**
###### **4.1 Настройка обучения**
+ Пользователь должен иметь возможность выбрать модель ***Ultralytics*** (YOLOv8, YOLOv9, YOLO11).


+ Пользователь должен иметь возможность задать гиперпараметры обучения (*epochs*, *batch size*, *image size*, *learning rate*).



###### **4.2 Запуск обучения**
+ После запуска обучения система должна начать процесс обучения модели с использованием локальной GPU.


+ Пользователь должен иметь возможность остановить процесс обучения.


+ При остановке обучения система должна сохранить текущие результаты и чекпоинт модели.



###### **4.3 Отображение результатов**
+ В процессе обучения система должна обновлять значения метрик после каждой эпохи.


+ Система должна отображать метрики обучения (loss, mAP, precision, recall).


+ После завершения обучения система должна сохранить обученную модель и сделать её доступной для скачивания.



##### **5. Трекинг экспериментов**
+ При каждом запуске обучения система должна сохранять параметры эксперимента.


+ Система должна сохранять полученные метрики и путь к обученной модели.


+ Пользователь должен иметь возможность просмотреть историю запусков.


+ Пользователь должен иметь возможность сравнить метрики нескольких экспериментов.



##### **6. Визуализация результатов**
+ Пользователь должен иметь возможность загрузить изображение для проверки модели.


+ После выполнения инференса система должна отобразить предсказанные bounding box на изображении.


+ Пользователь должен иметь возможность визуально проанализировать результаты предсказания.


### **Нефункциональные требования**
#### **1. Атрибуты качества**
##### **Производительность**
+ Система должна поддерживать работу с датасетами объёмом до 20 000 изображений.


+ Обучение модели должно выполняться с использованием локальной GPU.


+ Интерфейс не должен блокироваться во время выполнения длительных операций (обучение, предобработка).


##### **Надёжность**
+ При сбое во время обучения система должна сохранять текущие результаты эксперимента.


+ Данные датасета не должны теряться при создании новой версии.


##### **Воспроизводимость**
+ Система должна обеспечивать возможность повторного запуска эксперимента с сохранёнными параметрами.



#### **2. Требования к внешним интерфейсам**
+ Система должна обеспечивать взаимодействие с библиотекой ***Ultralytics*** для обучения моделей.


+ Система должна обеспечивать поддержку формата разметки ***YOLO***.


+ Система должна обеспечивать использование локальной CUDA-совместимой GPU.



#### **3. Ограничения проектирования и реализации**
+ Приложение должно работать локально и быть рассчитано на одного пользователя.


+ Система не должна требовать регистрации и авторизации пользователей.


+ Система не должна использовать централизованную базу данных для хранения изображений.


+ Хранение датасетов и моделей должно осуществляться файловым способом.


### Сценарии использования

### *Use Case 1 — Создание и управление датасетом*
**Действующее лицо:** Пользователь  
**Предусловие**:
Пользователь открыл приложение CV Studio.  
#### **Основной сценарий:**
1. Пользователь нажимает на кнопку создания проекта
2. Пользователь вводит название проекта.
3. Пользователь нажимает кнопку “Загрузить”
4. Пользователь выбирает папку с image/labels с корректным форматом кадров (jpg) с локального устройства.
5. Система создаёт новый датасет и отображает экран управления датасетом.
6. По желанию, пользователь может указать разбиение выборки на train/val
7. Пользователь нажимает на кнопку “сохранить”, система разбивает датасет

#### Альтернативный сценарий:
+ Пользователь загружает видео, система автоматически разбивает его на кадры
+ Пользователь отменяет загрузку файлов.
+ Система уведомляет об ошибке при некорректной структуре датасета.
+ Пользователь закрывает проект без сохранения изменений.


**Результат:**
Создан датасет с корректной структурой и разбивкой на выборки.

### *Use Case 2 — Выполнение аугментации датасета*
**Действующее лицо:** Пользователь
**Предусловие:**
Открыт существующий датасет.
#### Основной сценарий:
1. ользователь переходит в раздел «Аугментации».
2. Система отображает предпросмотр исходного изображения.
3. Пользователь настраивает параметры (Resize, Blur, Noise, …)
4. Система отображает результат в окне предпросмотра.
5. Пользователь нажимает «Применить ко всем».


#### Альтернативный сценарий:
+ Пользователь нажимает «Создать новую версию», система создаёт новую версию датасета.
+ Пользователь отменяет изменения.


**Результат:**
Создана новая версия датасета с применёнными аугментациями.

### *Use Case 3 — Разметка изображений*
**Действующее лицо:** Пользователь  
**Предусловие:**
Загружен датасет с изображениями.  
#### Основной сценарий:
1. Пользователь открывает раздел «Разметка».
2. Система отображает коллекцию изображений.
3. Пользователь открывает изображение.
4. Пользователь создаёт bounding box / polygon.
5. Пользователь выбирает класс объекта.
6. Пользователь сохраняет разметку.
7. Система обновляет файл разметки (формат .txt)
8. Пользователь переходит к следующему изображению.
#### Альтернативный сценарий:
+ Пользователь удаляет или редактирует bounding box / polygon.
+ Пользователь закрывает изображение без сохранения
+ Возникает ошибка сохранения файла.


**Результат:**
Изображения размечены и сохранены в формате YOLO.

### *Use Case 4 — Обучение модели и анализ результатов*
**Действующее лицо:** Пользователь

**Предусловие:**
+ Существует размеченный датасет с корректной структурой и разбиением на train/val.
+ В системе доступны ресурсы для запуска обучения.



#### Основной сценарий:
1. Пользователь переходит в раздел «Обучение».
2. Пользователь выбирает датасет
3. Пользователь выбирает архитектуру модели.
4. Пользователь задаёт гиперпараметры (Epochs, Batch size, Image size, Learning rate)
5. Пользователь нажимает «Запустить обучение».
6. Система добавляет задачу в очередь.
7. Система запускает процесс обучения.
8. Система отображает прогресс выполнения.
9. Система отображает текущие метрики (loss, accuracy и др.).
10. После завершения обучения система сохраняет модель и параметры запуска.
11. Система автоматически добавляет запуск в список экспериментов.
12. Пользователь переходит в раздел анализа.
13. Пользователь выбирает один или несколько экспериментов.
14. Пользователь нажимает «Сравнить».
15. Система отображает сравнительные метрики и графики.



#### Альтернативные сценарии:
##### A1 — Остановка обучения
+ Пользователь останавливает процесс.
+ Система сохраняет промежуточный чекпоинт.
##### A2 — Недоступность GPU
+ Система уведомляет пользователя.
+ Обучение не запускается.


##### A3 — Ошибка во время обучения
+ Система фиксирует ошибку.
+ Процесс завершается с уведомлением.


##### A4 — Нет завершённых экспериментов
+ Система уведомляет пользователя.
+ Сравнение недоступно.



**Результат:**
+ Модель обучена.
+ Параметры эксперимента сохранены.
Пользователь получил возможность проанализировать и сравнить результаты обучения.


## **Сквозной сценарий работы пользователя**
**Описание:**
 Сценарий демонстрирует последовательное использование UC-1 → UC-4 для получения обученной модели.

**Основной поток:**  
1. Пользователь выполняет UC-1.


2. Пользователь выполняет UC-3.


3. Пользователь выполняет UC-2.


4. Пользователь выполняет UC-4.


5. Пользователь получает обученную модель и результаты сравнения.



**Результат:**
Пользователь прошёл полный ML-цикл в рамках одной системы без перехода в сторонние инструменты.
